# RSA JWT Key Loading Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the shared JWT secret with RSA signing and validation, loading the key pair once at startup and keeping the rest of the auth flow unchanged.

**Architecture:** Keep the config layer responsible only for key paths and validation, keep a small startup helper in `cmd` responsible for loading and parsing PEM files once, and keep the token adapter responsible only for signing and verifying with parsed RSA keys. The `ports.TokenService` contract should remain unchanged so login, logout, middleware, and websocket call sites do not need behavioral changes.

**Tech Stack:** Go 1.26+, `github.com/golang-jwt/jwt/v5`, `crypto/rsa`, `crypto/x509`, `encoding/pem`, Echo, Viper, Testify, GoMock.

## Global Constraints

- JWTs are signed with RSA instead of a shared secret.
- The server loads RSA keys once at startup and fails fast on bad key material.
- JWT config uses key paths instead of a secret string.
- The existing auth flow continues to work without call-site churn.
- Tests cover config, startup failures, and RSA token verification.

```go
type jwtTokenService struct {
	privateKey       *rsa.PrivateKey
	publicKey        *rsa.PublicKey
	accessExpiration time.Duration
}

func NewJWTTokenService(privateKey *rsa.PrivateKey, publicKey *rsa.PublicKey, accessExpiration time.Duration) ports.TokenService {
	return &jwtTokenService{
		privateKey:       privateKey,
		publicKey:        publicKey,
		accessExpiration: accessExpiration,
	}
}

accessToken, err := jwt.NewWithClaims(jwt.SigningMethodRS256, accessClaims).SignedString(s.privateKey)

token, err := jwt.Parse(tokenString, func(token *jwt.Token) (interface{}, error) {
	if _, ok := token.Method.(*jwt.SigningMethodRSA); !ok {
		return nil, domain.ErrUnauthorized
	}
	return s.publicKey, nil
})
```

# Research: RFC 9457 Problem Details Migration

**Feature**: 003-rfc9457-error-format
**Date**: 2026-07-14

## 1. RFC 9457 Standard Structure

### Decision

Adopt RFC 9457 (STD 97) Problem Details for HTTP APIs as the error response format.

### Rationale

RFC 9457 is the Internet Standard (STD 97) that obsoletes RFC 7807. It defines a standard JSON object for expressing problem details in HTTP API error responses. The format is widely adopted (Spring Boot ProblemDetail, ASP.NET ProblemDetails, Django REST Framework, etc.) and enables off-the-shelf client libraries to parse errors without custom code.

### Standard Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string (URI) | Yes | A URI reference that identifies the problem type. Use `about:blank` when the type is unknown/generic. |
| `title` | string | Yes | A short, human-readable summary of the problem type. Should not change between occurrences (except for localization). |
| `status` | number | No | The HTTP status code generated by the server for this occurrence. |
| `detail` | string | No | A human-readable explanation specific to this occurrence of the problem. |
| `instance` | string | No | A URI reference identifying the specific occurrence of the problem (e.g., request path). |

### Extension Members (RFC 9457 §3.2)

The standard explicitly allows additional members. Our extensions:

| Extension | Type | Description |
|-----------|------|-------------|
| `code` | string | Machine-readable error code (`NOT_FOUND`, `INVALID_ARGUMENT`, etc.). Stable across releases. |
| `request_id` | string | Correlation ID matching the `X-Request-Id` header. |
| `errors` | array | Per-field validation failures. Only present when `status` is 400 and validation errors exist. |

### Content-Type

`application/problem+json` (RFC 9457 §4.1). This replaces the current `application/json` for error responses only.

### Alternatives Considered

- **Keep custom `{"error": {...}}` format**: Rejected — violates the user's explicit request for RFC compliance and misses interoperability benefits.
- **RFC 7807**: Rejected — it is obsolete. RFC 9457 is the current Internet Standard with identical structure.
- **JSON:API error format**: Rejected — more complex, less widely adopted for simple REST APIs, and adds unnecessary nesting (`{"errors": [{"status": ..., "code": ..., ...}]}`).

## 2. Type URI Generation

### Decision

Use a configurable base URL with a static slug suffix. Static mapping stored in `ErrorMapping`.

### Rationale

RFC 9457 §3.1.1 recommends that the `type` URI resolve to human-readable documentation about the problem type. By using a configurable base URL, we can point to different documentation hosts per environment (dev docs, production docs).

### Mapping Strategy

Each domain error type maps to a URI slug:

| Domain Error | Status | Type URI Slug | Title |
|---|---|---|---|
| `ErrUserNotFound` | 404 | `not-found` | Not Found |
| `ErrPostNotFound` | 404 | `not-found` | Not Found |
| `ErrCommentNotFound` | 404 | `not-found` | Not Found |
| `ErrNotificationNotFound` | 404 | `not-found` | Not Found |
| `ErrNotFollowing` | 404 | `not-found` | Not Found |
| `ErrNotLiked` | 404 | `not-found` | Not Found |
| `ErrInvalidCredentials` | 401 | `unauthenticated` | Unauthorized |
| `ErrUnauthorized` | 401 | `unauthenticated` | Unauthorized |
| `ErrValidationFailed` | 400 | `invalid-argument` | Bad Request |
| `ErrInvalidPost` | 400 | `invalid-argument` | Bad Request |
| `ErrInvalidNotification` | 400 | `invalid-argument` | Bad Request |
| `ErrCannotFollowSelf` | 422 | `invalid-argument` | Unprocessable Entity |
| `ErrCannotUnfollowSelf` | 422 | `invalid-argument` | Unprocessable Entity |
| `ErrEmailAlreadyRegistered` | 409 | `already-exists` | Conflict |
| `ErrAlreadyFollowing` | 409 | `already-exists` | Conflict |
| `ErrAlreadyLiked` | 409 | `already-exists` | Conflict |
| `ErrUsernameAlreadyTaken` | 409 | `already-exists` | Conflict |
| `ErrInternalServer` | 500 | *(about:blank)* | Internal Server Error |

**Note**: Multiple sentinel errors map to the same `type` URI slug and `title` because they represent the same HTTP-level problem category. The `detail` field differentiates specific occurrences.

### Default Base URLs

- **Production**: `https://api.project-one.dev/errors/`
- **Development**: `http://localhost:8080/errors/`
- **Config key**: `app.error_type_base_url` (empty defaults to development URL)

### Unknown/Unmapped Errors

Use `about:blank` with title "Internal Server Error" and status 500. This is the RFC 9457 §3.1 recommendation for generic errors where no more specific type URI is known.

### Alternatives Considered

- **Per-error unique URIs** (e.g., `/errors/user-not-found` vs `/errors/post-not-found`): Rejected — adds complexity with no consumer benefit. The `detail` field already carries the specific message.
- **Hardcoded URI in code**: Rejected — breaks environment portability. Config-driven URIs allow different hosts per deployment.

## 3. Config Integration

### Decision

Add `ErrorTypeBaseURL` field to `config.AppConfig`.

### Rationale

The existing `AppConfig` struct in `internal/config/config.go` already holds environment-level settings (`port`, `env`). The error type base URL is an environment-level setting that varies between dev/staging/production.

### Config Structure Addition

```yaml
app:
  port: 8080
  env: "development"
  error_type_base_url: ""  # defaults to http://localhost:{port}/errors/
```

### Alternatives Considered

- **Separate config section**: Rejected — over-engineering for a single string setting.
- **Environment variable only**: Rejected — `config.yaml` is the project's canonical config source; env vars are secondary.

## 4. Frontend Error Parsing

### Decision

Update `web/lib/errors.ts` to parse `application/problem+json` responses. Extract `detail` for user-facing messages, `status` for HTTP-aware decisions, `errors` array for form field validation.

### Rationale

The frontend currently parses `{"error": {"message": "..."}}` from any JSON error response. The new format requires extracting from top-level fields instead of a nested `error` object. The `ApiError` class gains a `code` field and `errors` array.

### Updated Error Extraction

| Current | RFC 9457 |
|---------|----------|
| `errorData.error.message` → user message | `errorData.detail` → user message |
| `response.status` → HTTP code | `errorData.status` → HTTP code (prefer body field, fallback to response) |
| `errorData.error.code` → machine code | `errorData.code` → machine code (extension) |
| `errorData.error.details` → field errors | `errorData.errors` → field errors (extension) |

### Alternatives Considered

- **New error parsing library**: Rejected — the change is small (field name remapping); a new dependency is unnecessary.
- **Backward-compatible parsing** (try both formats): Rejected — the spec requires the new format exclusively. Old format is removed, not deprecated.

## 5. Swagger/OpenAPI Documentation

### Decision

Regenerate Swagger docs using existing swaggo annotations, updated to reference `application/problem+json` and the RFC 9457 schema.

### Rationale

The project uses swaggo for API documentation generation. The handler annotations (`@Failure`, `@Success`) already document error responses. We update the referenced response type and regenerate with `make docs`.

### Schema Approach

Define a single `dto.ProblemDetail` struct annotated with swaggo tags, referenced by all endpoint failure annotations. This ensures consistency across all documented endpoints without duplicating schema definitions.

### Alternatives Considered

- **Manual swagger.json editing**: Rejected — fragile and not reproducible. swaggo annotations are the source of truth.
- **OpenAPI 3.1 separate schema file**: Rejected — the project uses swaggo's annotation-based approach. Introducing a separate schema file would break the existing docs workflow.

## 6. Backward Compatibility

### Decision

No backward compatibility. The format changes atomically.

### Rationale

The existing error format is consumed only by the project's own frontend, which is updated in the same feature. No external API consumers depend on the current format. A clean cut avoids dual-format complexity.

### Migration Impact

- **Backend**: All error responses change format immediately upon deployment.
- **Frontend**: Must be deployed synchronously with the backend to avoid parsing failures.
- **Database**: No changes (format is transport-only).
- **Tests**: All error-related tests updated to assert RFC 9457 structure.
